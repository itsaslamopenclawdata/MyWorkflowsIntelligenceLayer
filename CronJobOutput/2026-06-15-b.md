### Step 3 — Build the write-off analyzer (`~/WriteOffPilot/analyzer.py`)

- **WHAT:** A runnable Python pipeline that takes a CSV of EOB write-off rows (or an Excel export from Open Dental / Dentrix / Eaglesoft) and emits a 1-page PDF audit: total written off, breakdown by reason code, breakdown by insurance payer, top 10 highest-write-off claims, and a written paragraph ("you wrote off $X this quarter, $Y is recoverable by resubmission — that's $Z/year if held constant"). Also emits an Excel scoring sheet for the office manager to drill into.
- **WHO:** `Hermes`
- **HOW:** Create `~/WriteOffPilot/` with 3 files:
  - `analyzer.py` — orchestrator. Sub-steps:
    1. `def load_data(input_path: Path) -> pd.DataFrame` → reads CSV or `.xlsx` (Open Dental exports both), normalizes column names to `claim_id, date, payer, procedure_code, billed_amount, paid_amount, writeoff_amount, writeoff_reason_code, writeoff_reason_text`. Maps common reason codes (CO-16, CO-29, CO-45, CO-97, PR-1, PR-2, PR-3) to friendly labels ("Additional Information", "Time Limit", "Charge Exceeds Fee Schedule", "Benefit Included in Another Service", "Deductible", "Coinsurance", "Copay").
    2. `def categorize_writeoffs(df: pd.DataFrame) -> pd.DataFrame` → adds a `recovery_likelihood` column (HIGH = denied for missing info that resubmission fixes = CO-16, CO-29, CO-97; MEDIUM = coordination of benefits, eligibility, timely filing = CO-29, CO-31; LOW = patient responsibility, contractual write-off = PR-1, CO-45). High-likely ones are the "recoverable" bucket.
    3. `def summarize(df: pd.DataFrame) -> dict` → `{"total_writeoff": float, "recoverable_writeoff": float, "recoverable_annualized": float, "by_payer": dict, "by_reason": dict, "top10_claims": list[dict]}`.
    4. `def render_pdf_audit(summary: dict, practice_name: str) -> Path` → WeasyPrint, 1-page template `audit_template.html` (header with practice name + report date; 3 KPI tiles: total writeoff, recoverable, annualized; horizontal bar chart of top 5 reason codes; top-10 table; a 1-paragraph narrative).
    5. `def emit_excel_audit(df: pd.DataFrame, summary: dict, output: Path) -> Path` → `df.to_excel(sheet_name="All Writeoffs")` + summary KPIs on a second sheet.
    6. `def main()` → argparse with `--input`, `--practice-name`, `--output-dir`. End-to-end: `load → categorize → summarize → render → emit`.
  - `audit_template.html` — Jinja-style HTML, 1 page, with placeholders.
  - `README.md` — usage: `python3 analyzer.py --input sample_eobs.csv --practice-name "Sample Family Dental" --output-dir ./out`.
  - `sample_eobs.csv` — a synthetic 50-row sample EOB file (generated in the smoke test) so the smoke test can run without a real client.

## 6. Failure Modes and Auto-Recovery

- **Failure:** `OPENAI_API_KEY` not set in env (Step 1 fail).
  - **Detection:** `python3 -c "import os; assert os.environ.get('OPENAI_API_KEY')"` raises AssertionError.
  - **Auto-recovery:** none. ASLAM-GATE Step 1.1. Aslam pastes the key (1 min).
  - **Escalate to Aslam only if:** key paste is required AND Step 6.1 ASLAM-GATE also needs running — bundle them into one 2-min trip.

- **Failure:** `pymupdf` not installed (Step 1 fail).
  - **Detection:** `python3 -c "import pymupdf"` raises ImportError.
  - **Auto-recovery:** Hermes runs `pip install --user pymupdf` (or in a venv if PEP 668 blocks system install). Re-runs the import check. If still failing, the analyzer falls back to `pdfplumber` (already a likely-installed alternative) by switching the import line to `try: import pymupdf as pdf; except ImportError: import pdfplumber as pdf` and adjusting the table-extraction calls. The PDF *rendering* path (WeasyPrint) is unaffected — `pymupdf` is only needed for parsing existing PDFs, which the analyzer doesn't do for EOBs that are already CSV/XLSX. The smoke test in Step 3 will still pass even if `pymupdf` is missing because the sample_eobs.csv is CSV input.
  - **Escalate to Aslam only if:** both `pymupdf` and `pdfplumber` fail to install (network down on this box).

- **Failure:** B2B scout dental 403 / 502 from YouTube search (Step 1 partial).
  - **Detection:** `dental_scout.log` shows `403 Forbidden` or `502 Bad Gateway`.
  - **Auto-recovery:** The scout's existing graceful-degradation writes the report header + "queries failed" placeholder. The report file still exists at `B2BRealData/B2BScout-2026-06-15-dental.md` with > 5 lines, satisfying the verify check. The buyer pool is the freshly-scraped CSV from Step 2, not the scout report.
  - **Escalate to Aslam only if:** 0 dental opportunities surface from any source for 3 consecutive runs.

- **Failure:** Cold email send blocked (SMTP rejected, Gmail auth fail, no `cold-outbound-sender.py`, no `email` skill).
  - **Detection:** `~/WriteOffPilot/logs/send-2026-06-15.log` shows `sent_count=0` and the send script's stderr has `AuthenticationError` or `SMTPAuthenticationError`.
  - **Auto-recovery:** Hermes runs the last-resort send script path (the `python3 -c "...smtplib..."` block) with creds from `~/.hermes/smtp_creds.json`. If creds file missing, ASLAM-GATE Step 6.1 runs.
  - **Escalate to Aslam only if:** both Gmail SMTP and any other send path fail AND no other email skill is wired.

- **Failure:** Pipeline smoke test fails (no PDF or no Excel).
  - **Detection:** `test -f /tmp/wo_test_out/SmokeTest-audit.pdf` exits non-zero.
  - **Auto-recovery:** Hermes reads the traceback, fixes the most common cause (WeasyPrint missing system libs → swap to `reportlab` directly to render the PDF). Retries the smoke test up to 2 times. If still failing, falls back to plain-text + Excel only (skip the PDF render — the prospect still gets a useful artifact).
  - **Escalate to Aslam only if:** all PDF paths fail AND the prospect insists on PDF. In that case deliver a markdown report (`out/<slug>/audit.md`) and note the PDF will be regenerated when libs are fixed.

- **Failure:** 0 replies after 7 days.
  - **Detection:** `wc -l ~/WriteOffPilot/logs/replies-2026-06-15.jsonl` is 0 on Day 7.
  - **Auto-recovery:** Hermes sends a second wave to 20 more prospects (same template, broader state filter including NC, GA, MI, OH). Updates the prospect list. If still 0 by Day 14, pivots to a Gumroad-packaged version of the same analyzer (template + sample EOBs + scoring rubric + sample audit PDF) at $19.
  - **Escalate to Aslam only if:** the second wave also fails and Aslam wants to manually call the top-3 prospects himself.

- **Failure:** HIPAA / data-handling concern (prospect's actual EOB data contains PHI).
  - **Detection:** prospect sends real claim data with patient names or DOBs in the rows.
  - **Auto-recovery:** Hermes runs the analyzer LOCALLY (not in cloud) and never logs raw claim rows. The output PDF aggregates by reason code and total $ — no individual patient rows. Hermes also strips `claim_id` and any DOB fields from the input DataFrame before summarizing. If the prospect asks about HIPAA, Hermes sends the 1-paragraph explanation: "All processing runs on our local machine — your patient data never leaves our system. The output report shows aggregate totals by reason code and insurance payer, no individual patient rows. If we proceed to a paid subscription, we add a BAA clause to the contract and run everything on an encrypted local volume."
  - **Escalate to Aslam only if:** the prospect requires a formal BAA (which we don't have ready) — Aslam either signs the standard BAA template or refers the prospect to the per-call one-time analysis model (no subscription, no stored data).

- **Failure:** Aslam is offline / Telegram approval doesn't come.
  - **Detection:** 2-hour timeout on Step 5 approval, no reply in `~/.hermes/gumroad_approval_state.json`.
  - **Auto-recovery:** Hermes sends a one-line Telegram ping: "Approval still pending on Dental Write-Off outreach. Will hold send until ACK. Reply APPROVE or EDIT: <change>." Waits another 4 hours. If still no reply, Hermes saves the draft email to `~/WriteOffPilot/outreach/drafts/pending-approval.md` and exits Step 5 in `awaiting-approval` state. The cron will retry tomorrow.
  - **Escalate to Aslam only if:** Aslam explicitly tells Hermes to never block on Telegram approvals (then Hermes auto-approves low-stakes drafts via the `decision-making` skill).
